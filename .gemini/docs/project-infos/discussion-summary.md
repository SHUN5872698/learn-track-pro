## これまでの議論で確定した方針

### 1. 転職準備の段階的改善アプローチ

**要点**: アプリケーションロジック理解 → ドキュメント整備 → CI/CD構築 → 機能開発の順序で進める

**判断基準**: 確実性の原則、品質担保の基盤構築、継続的な学習姿勢の実証

### 2. Phase 1：アプリケーションロジック理解が最優先

**要点**: AI依存で実装した箇所（Chart.js統計計算、API設計、認証フロー、状態管理、コンポーネント間データフロー）の完全理解

**判断基準**: README作成時に正確な説明が必要、面接対策として必須、ドキュメント添削時の整合性確保、履歴書作成時の深堀りが前提

### 3. Phase 2：ドキュメント整備が第二優先

**要点**: 転職活動開始の前提条件を満たす（README、ドキュメント統一、履歴書・職務経歴書作成）

**判断基準**: タイムリミット（2026年1月までに書類完成必須）、確実性の原則（技術的リスクゼロ）、採用担当者はまずREADMEとドキュメントを見る

**実施タスク**: ドキュメント添削 → README作成 → 履歴書・職務経歴書作成

### 4. Phase 3：CI/CD構築がPhase 4より優先される理由

**要点**: 品質を担保した継続的開発の基盤構築、本番稼働中のアプリケーションへの変更には自動テストが必須

**判断基準**: 機能追加時のデグレード防止、企業が求めるスキル（CI/CD構築スキル：★★★★★）、開発効率向上、段階的改善の実践

### 5. Phase 4：機能開発は転職活動と同時進行

**要点**: 継続的な学習姿勢の実証、転職活動の待ち時間を開発に充てる

**判断基準**: 優先度が低い（あったら便利レベル）、転職活動に支障をきたさない、CI/CDがあるため安心して機能追加可能

### 6. 転職準備ドキュメントの記述方針

**要点**: 「何をしたか」ではなく「何が書いてあるか」にフォーカス、変更・修正などのアクションではなくコンテンツの内容を記載

**判断基準**: 後から読み返したときに内容が理解できることが重要

### 7. 所要時間見積もりを記載しない方針

**要点**: 根拠のない見積もりは削除、現段階では無価値

**判断基準**: 実際の作業時間を記録していない場合は見積もりを書かない、リアルな作業時間のみ記載

### 8. 包括的ドキュメントの管理方針（従来）

**要点**: 3つ以上の詳細ドキュメントがある場合、包括的ドキュメントを作成してDB表示件数を削減

**判断基準**: 視認性向上、全体像把握、認知負荷軽減

### 9. Git戦略とブランチ管理（従来）

**ブランチ構成**: main（本番専用）、develop（開発・統合）、feature（個別機能開発）

**.gitattributes による環境別ファイル保護**: config/cors.php merge=ours、.gitignore merge=ours

**運用ルール**: ローカル設定変更はdevelopでコミット、本番設定変更はmainで直接コミット

## やってはいけないこと

1. SSHパスワード認証を有効化
2. rootで直接ログイン許可
3. MySQLのDocker化
4. mainでローカル用設定をコミット
5. .gitattributesなしで環境別ファイル管理
6. git clone後にstorage/権限設定をしない
7. 詳細ドキュメントを子ページ化
8. 実際の構築時間を架空の「理想的な所要時間」で記載
9. Phase番号を重複させる
10. 「何をしたか」にフォーカスした記録（「何が書いてあるか」にフォーカスすべき）

## 技術選択と教訓

### 重要な技術選択の判断

1. **段階的改善アプローチ**: 一気に全てをやろうとせず、Phase 1→2→3→4の順序で確実に進める
2. **転職活動の現実的なタイムライン**: 2026年1月開始、4月入社を目標に逆算したスケジュール
3. **AI依存箇所の理解を最優先**: 面接で「AI依存度」を見抜かれるリスク回避

### 重要な教訓

1. **確実性の原則**: 現在の実績を確実に記録することが最優先、不確実な要素（CI/CD実装可否）は後回し
2. **採用担当者視点の重要性**: GitHubを見る際、最初に見るのはREADMEとドキュメント
3. **タイムリミットの厳守**: 書類が完成していないと転職活動が開始できない
4. **ポートフォリオ価値の優先順位**: CI/CD構築スキル > コード最適化 > 新機能追加
