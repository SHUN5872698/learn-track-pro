## これまでの議論で確定した方針

### 1. 転職準備の段階的改善アプローチ

**要点**: アプリケーションロジック理解 → ドキュメント整備 → CI/CD構築 → 機能開発の順序で進める

**判断基準**: 確実性の原則、品質担保の基盤構築、継続的な学習姿勢の実証

### 2. Phase 1：アプリケーションロジック理解が最優先

**要点**: AI依存で実装した箇所（Chart.js統計計算、API設計、認証フロー、状態管理、コンポーネント間データフロー）の完全理解

**判断基準**: 面接対策として必須、README作成時に正確な説明が必要、ドキュメント添削時の整合性確保

### 3. アプリケーションロジック理解の方針

**要点**: 実装内容（What）ではなく、設計判断の理由（Why）を中心に理解

**判断基準**: 企業の技術面談で説明できるレベルを目標

### 4. ドキュメント整理の方針

**要点**: 読み返しやすさを最優先、GeminiCLI冗長回答を簡潔な箇条書きに統一

**判断基準**: Backend理解と同じフォーマットに統一、理解の定着率向上

### 5. 添削プロンプトの設計方針

**2種類のプロンプト**:

- ドキュメント整理用：Backend理解を参考フォーマットとして指定
- 理解評価用：簡潔さ・Why重視・正確性の3観点で評価

**判断基準**: マスターデータ以外（ユーザー認証等）にも適用可能な汎用構造

### 6. Phase 2：ドキュメント整備が第二優先

**要点**: 転職活動開始の前提条件を満たす（README、ドキュメント統一、履歴書・職務経歴書作成）

**判断基準**: タイムリミット（2026年1月までに書類完成必須）、採用担当者はまずREADMEとドキュメントを見る

### 7. Phase 3：CI/CD構築がPhase 4より優先される理由

**要点**: 品質を担保した継続的開発の基盤構築、本番稼働中のアプリケーションへの変更には自動テストが必須

**判断基準**: 機能追加時のデグレード防止、企業が求めるスキル（CI/CD構築スキル：★★★★★）

### 8. 包括的ドキュメントの管理方針

**要点**: 3つ以上の詳細ドキュメントがある場合、包括的ドキュメントを作成してDB表示件数を削減

**判断基準**: 視認性向上、全体像把握、認知負荷軽減

### 9. 詳細ドキュメントを子ページ化しない

**要点**: すべてDB管理下に独立配置、包括ドキュメントからメンションリンクで接続

**判断基準**: 親ページ削除時の誤削除リスク回避、検索性向上

### 10. Git戦略とブランチ管理

**ブランチ構成**: main（本番専用）、develop（開発・統合）、feature（個別機能開発）

**運用ルール**: ローカル設定変更はdevelopでコミット、本番設定変更はmainで直接コミット

詳細: 環境構築メモ⑥参照

### 11. VPS環境のファイル権限設定

**要点**: `storage/`と`bootstrap/cache/`は`www-data:www-data`、パーミッション775

**判断基準**: git clone直後に必ず設定することで時限爆弾的なPermission deniedエラーを防止

詳細: 環境構築メモ⑤参照

### 12. デプロイ先・構成

**結論**: Xserver VPS 2GB（月1,150円）、LaravelのみDocker化

**理由**: 予算制約、メモリ2GB環境での安定性優先

## やってはいけないこと

1. SSHパスワード認証を有効化
2. socket activation有効でポート変更
3. MySQLのDocker化
4. mainでローカル用設定をコミット
5. .gitattributesなしで環境別ファイル管理
6. git clone後にstorage/権限設定をしない
7. 詳細ドキュメントを子ページ化
8. 実際の構築時間を架空の「理想的な所要時間」で記載
9. 「何をしたか」にフォーカスした記録（「何が書いてあるか」にフォーカスすべき）
10. アプリケーションロジック理解で実装内容のみにフォーカス

## 技術選択と教訓

### 重要な技術選択の判断

1. **段階的改善アプローチ**: Phase 1→2→3→4の順序で確実に進める
2. **転職活動の現実的なタイムライン**: 2026年1月開始、4月入社を目標に逆算
3. **AI依存箇所の理解を最優先**: 面接で「AI依存度」を見抜かれるリスク回避
4. **アプリケーションロジック理解フレームワーク**: データフロー把握→Backend→Frontend→統合理解の4ステップ

### 重要な教訓

1. **確実性の原則**: 現在の実績を確実に記録することが最優先
2. **採用担当者視点の重要性**: GitHubを見る際、最初に見るのはREADMEとドキュメント
3. **Why重視の理解**: 実装内容より設計判断の理由が面接では重要
4. **読み返しやすさ優先**: 冗長回答簡潔化で理解定着率向上
