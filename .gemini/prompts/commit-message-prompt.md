# コミットメッセージ生成ルール

## 必須の出力形式

```
type: 変更内容の要約（30-50文字）

変更の全体的な説明を1行で記述（50-70文字）
- 具体的な変更点1（体言止め）
- 具体的な変更点2（体言止め）
- 具体的な変更点3（体言止め）
```

## typeの判定基準（厳密に適用）

| type | 使用条件 | 例 |
|------|---------|-----|
| feat | ユーザーに見える新機能・機能追加 | 新しいボタン、新しい画面、新しい機能 |
| fix | バグ・不具合の修正 | エラー修正、表示崩れ修正 |
| refactor | 内部構造の改善（動作は同じ） | コンポーネント化、重複削除、整理 |
| chore | 上記以外（設定、削除、依存関係） | 不要ファイル削除、パッケージ更新 |

## 文体ルール（必ず守る）

1. **タイトル**: 「〜を○○」の形式
   - ✅ 良い例: `汎用ボタンコンポーネントを導入`
   - ❌ 悪い例: `汎用ボタンコンポーネントの導入`

2. **説明文**: 「〜を○○」で終わる完全な文
   - ✅ 良い例: `アプリケーション全体で使用できる汎用ボタンコンポーネントを新規追加`

3. **箇条書き**: 体言止めまたは「〜を○○」
   - ✅ 良い例: `UIの一貫性と保守性を大幅に向上`
   - ❌ 悪い例: `UIの一貫性と保守性が向上しました`

## 実例テンプレート

### feat の場合
```
feat: [新機能名]を[動作]

[対象範囲]で[利用目的]の[機能名]を[動作]
- [追加した要素や機能]
- [提供する価値や効果]
- [技術的な改善点]
```

### refactor の場合
```
refactor: [対象]を[改善内容]

[変更対象]を[新しい方式]に[変更内容]
- [具体的な変更箇所]
- [削減した重複や改善した点]
- [得られた効果]
```
#### 成功例（これを真似する）
```
feat: 汎用ボタンコンポーネントを導入

アプリケーション全体で使用できる多様なスタイルの汎用ボタンコンポーネントを新規追加
- プライマリ、セカンダリ、危険、アイコンなど多様なバリアント
- ボタンのスタイルと動作を一元管理
- UIの一貫性と保守性を大幅に向上
```

```
refactor: 既存ボタンを汎用コンポーネントに置き換え

既存のボタン要素を汎用ボタンコンポーネントに統一的に置き換え
- モーダル、カード、フォーム、認証ページなどで適用
- コードの重複を削減し保守性を向上
- UIの統一化により一貫性を確保
```


## 必須チェックリスト（生成後に自己確認）

- [ ] タイトルは50文字以内か
- [ ] 説明文は1行で完結しているか
- [ ] 箇条書きは3-5項目か
- [ ] 体言止めで統一されているか
- [ ] **ファイルに出力したか** ← ⚠️最重要

## 最終アクション（絶対に実行）

1. コミットメッセージを生成
2. チェックリストで確認
3. **必ず** `.gemini/outputs/commit-message-for-docs.md` に出力
4. 出力完了後、タスク終了

**重要**:
- ファイル出力なしでタスクを終了しない
- 追加の提案や説明は不要
- **出力したら「ファイルに出力しました」と報告して終了**
